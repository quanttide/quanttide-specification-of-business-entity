# 工程标准资产规范

本文档定义工程标准类数字资产的结构和元数据规范。

## 定义

工程标准资产是用于规范工程实践、流程和方法的标准化文档。

## 分类

工程标准资产按功能领域分类：

### 开发规范

- 编码规范
- API设计规范
- 数据库设计规范
- 架构设计规范

### 流程规范

- 版本发布规范
- 代码审查规范
- 测试规范
- 部署规范

### 文档规范

- 文档格式规范
- 写作风格指南
- 文档结构规范
- API文档规范

### 运维规范

- 监控规范
- 日志规范
- 备份规范
- 安全规范

## 结构

### 目录结构

```
specification/
├── default/          # 默认工程标准
├── devops/           # DevOps工程标准
├── docs/             # 文档工程标准
├── meta/             # 元工程标准
├── think/            # 思维工程标准
└── write/            # 写作工程标准
```

### 文件命名

- 使用小写字母
- 单词间用下划线分隔
- 文件名应简洁明确
- 示例：`format.md`、`release.md`

### 元数据

工程标准文档应包含以下元数据：

```yaml
---
title: 标准名称
version: 版本号
status: 状态（draft/review/approved）
created: 创建日期
updated: 更新日期
---
```

## 内容规范

### 必需章节

1. **概述**：说明标准的目的和适用范围
2. **规范内容**：具体的标准要求
3. **参考文档**：引用的外部标准或资料

### 可选章节

- 定义：术语和概念定义
- 背景：标准制定的背景和原因
- 示例：具体应用示例
- 检查清单：合规检查项

### 格式要求

- 遵循文档格式规范（参见docs/format.md）
- 使用Markdown格式
- 标题层级不超过三级
- 代码块必须标注语言类型

## 版本管理

### 版本号

遵循语义化版本规范：`主版本.次版本.修订号`

- **主版本**：不兼容的重大修改
- **次版本**：向后兼容的功能新增
- **修订号**：向后兼容的问题修复

### 更新流程

1. 修改标准内容
2. 更新版本号和日期
3. 更新CHANGELOG
4. 提交并推送到仓库

## 合规性

### 审查要求

工程标准发布前应经过：

- 技术审查：技术可行性
- 业务审查：业务适配性
- 文档审查：文档规范性

### 实施要求

- 明确实施时间表
- 提供实施指南
- 建立反馈机制
- 定期评估更新
